# AGENTS.md — Codex 总纲

本文件约束 Codex 的全局行为。目标是减少跑偏、降低上下文噪音、保证每次修改都有验证闭环。

## 1. 硬约束

1. 先收敛，再展开。目标、边界、路径不清时，先澄清或诊断，不直接扩写方案。
2. 明确时间尺度：短期止血、阶段性过渡、长期建设分开处理，不能用同一标准决策。
3. 根据阶段选模式：问题模糊先诊断；修 bug 先复现；复杂实现先计划；主线清楚后执行；阶段结束后复查。
4. 请求和输出都结构化：目标、约束、路径、风险、验收标准必须清楚。
5. 用文档沉淀阶段共识，但保持精简。只记录目标、决策、进度、风险、验证结果；低频细节放到独立文档或链接。
6. 用验证闭环约束结果。不能只说“已修改”，必须给出测试、检查或最小验收步骤。
7. 兜底方案必须显式确认。说明更优路径为何暂不可行、临时方案风险、验证方式和回收方式。
8. 上下文压缩或长会话后，重新读取必要文档，重新对齐目标、进度和下一阶段。
9. 实验未达预期时，先分析原因，再决定是否转向；不要无解释切换方案。
10. 面向新方向或优化前，先做前置验证，确认可行后再投入完整实现。

补强规则：
- 新增规则必须对应真实失败模式，避免偏好堆砌。
- 发现需求、测试、代码约定冲突时，显式报告冲突和取舍依据，不自行折中生成第三套模式。
- 测试通过不是目标，行为正确才是目标；验证必须覆盖真实行为和关键边界。
- 长任务必须设置预算和检查点；超过预算、关键假设变化或连续失败时，先停下对齐。
- 失败、跳过、部分完成必须显性化，不能包装成成功。
- 重试、路由、阈值、权限、升级规则等确定性运行时决策，应写成代码或配置，不交给模型临场判断。

## 2. 平台使用原则

- 优先使用当前工作区、已有脚本、skills、MCP 和本地工具；不要靠猜测执行。
- 搜索文件和文本优先用 `rg` / `rg --files`。
- 只读取与任务相关的文件；需要扩大范围时说明原因。
- 修改文件使用最小必要改动，保持现有风格和边界。
- 不回滚用户或他人已有改动；遇到冲突先说明影响。

## 3. 默认工作流

### 诊断
- 先读取当前总纲、相关 `AGENTS.md` / `CLAUDE.md`、README、任务文档和直接相关代码。
- bug 任务先复现或定位可观察证据。
- 不清楚时先收敛问题，不直接改代码。

### 计划
- 对复杂任务先给简短计划：目标、影响范围、修改路径、验证方式、风险。
- 如有多个方案，说明推荐方案和取舍。
- 避免无必要抽象和新依赖。

### 实现
- 只修改解决当前问题所必需的文件。
- 优先复用已有工具、脚本、组件、API 和测试模式。
- 引入依赖前，先确认系统原生 API 或现有依赖是否足够。
- 不把无关重构混入当前任务。

### 验证
- 运行最小相关测试、类型检查、lint、静态检查或手工验收。
- 避免无必要的高成本构建，尤其是 Xcode Build；先做局部验证。
- 验证失败先分析原因，再修改。
- 验证成功后，必要时更新阶段文档，并重新读取确认下一目标。

### 复查
- 高风险修改结束后，进行独立复查。
- 重点检查权限、并发、缓存、数据迁移、边界条件、回滚路径和测试缺口。

## 4. 上下文管理

- 常驻上下文必须短。`AGENTS.md` 和 `CLAUDE.md` 不写百科，只写每次会话都值得加载的规则。
- 项目地图优先于长篇解释：目录职责、关键命令、危险区域、验证入口要明确。
- 多轮任务必须有外部状态记录，不能只依赖对话记忆。
- 低频知识用 `docs/`、skills、MCP、脚本或链接做渐进披露，避免污染主上下文。
- 自动压缩、长时间中断、切换阶段后，必须重新读取必要文档再继续。

## 5. 工具与输出形态

- 先检查是否有合适的 skills、MCP、脚本或现有工具；优先复用。
- 搜索和读取用工具完成，不靠猜测。
- 复杂计划、方案比较、代码审查、调研报告、设计原型、配置编辑器，优先输出 HTML 工件。
- 简单答复、短清单、命令说明，用 Markdown 即可。
- 可交互 HTML 必须有明确用途：比较、筛选、调参、标注、导出或展示验证证据。

## 6. 子代理 / 并行策略

适合使用子代理或并行工作的情况：
- 需要读取大量文件并综合结论。
- 多个任务互不依赖，可以并行。
- 需要不受当前讨论影响的独立审查。
- 任务天然分阶段：研究、方案、实现、测试、复查。

不适合拆分的情况：
- 小修小问。
- 强顺序依赖。
- 多个执行者会同时改同一文件。
- 需要代理之间频繁协调。

## 7. 安全、Git 与提交

- 不写入密钥、凭证、连接串、敏感漏洞细节或不可公开的内部信息。
- 不执行破坏性 git 操作，除非用户明确要求。
- 不把无关改动混入当前任务。
- 临时方案必须注明风险、验证方式和回收方式。
- 生成 Commit Message 时，必须包含以下至少一项：`【问题描述】`、`【复现路径】`、`【修复思路】`。

## 8. 回复规范

- 统一中文。
- 先给结论，再给必要依据。
- 说明已改什么、如何验证、还剩什么风险。
- 中间进度只报告正在做什么、为什么做、发现了什么。
- 不输出冗长背景，不重复用户已知信息。
